\chapter*{Preface}
\addcontentsline{toc}{chapter}{Preface}

\epigraph{\textit{Omne tulit punctum qui miscuit utile dulci, \\
lectorem delectando pariterque monendo.} \\
He who blends usefulness and pleasantness wins every
vote, at once delighting and instructing the reader.}%
{Horace - Ars Poetica (343-344)}\index{Horace}

When Horace postulates in his "Ars Poetica" (19 B.C.) that one of the
qualifications of a poet is to mix the usefulness with sweetness in order to
both delight and instruct the reader, little would he have known that this
would not only apply to the written art of poetry, but to the art of
programming as well. Taken from verse 343 of this poem is the Latin phrase
\textit{miscere utile dulci} that captures this principle. Abbreviated to \MUD,
is the project name that is being uses as the basis for naming various
projects. Included in this, is the \MUD Library.

\MUDLib is a general purpose application framework library that conceptualises
various aspects of \Cpp application development. A number of such frameworks
already exists and have been quite successful. So why is there a need for yet
another one? Well, there probably is not really a need for it, but I will
refer to the premise in the previous paragraph. It's intention is not only to
be useful and delightful for anyone who may find it suitable for their own
purposes, but equally so, it will be useful and delightful for the author
who created it. This includes the ever evolving \Cpp language constructs as
well as the need to abstract and combine many principles of application
development into a consistent single framework.

Let's take, as an example, the technology behind programming a TCP server and
client application. Back in 1983, the Berkeley Sockets API (application
programming interface) was introduced to define the functions to support
creating of sockets, establishing and accepting connections and sending and
receiving data from one endpoint to another. This API was adopted into the
POSIX (Portable Operating System Interface) standard and is supported on
almost any platform. For example, it is incorporated into UNIX systems and
their derivations, like Oracle Solaris, Linux and Apple Darwin, and also in
Microsoft Windows. Even to this day, applications written in the 80s would
still be able to build and run on modern operating systems.

Throughout the decades, advances has been made in all aspects of computer
science. The adoption of object-oriented programming is one of those advances
and led to the introduction of \Cpp and other programming languages. This has
created some a dilemma that the procedural API for handling sockets of POSIX
has to be mapped within an object-oriented framework. Some library providers
have adopted these concepts successfully and these have been widely adopted,
even to the point that they have been incorporated into \Cpp standards (like
some aspects of the Boost library).

However, with all these different libraries, there is no single abstraction of
such concepts which is a 'correct' one. An ACE (Adaptive Communication
Environment) socket differs in a Boost socket by their viewpoint on how a
socket can be represented. Although there is inevitably a great number of
overlap of functionality that each has to provide, the differences between them
and the design concepts they implemented, needs to be assessed to their
usefulness in the specific problem domain and which can be applied more
effectively.

Another aspect of application framework libraries is that there are many
libraries that focus on a certain key feature, but there are only a few that
cater an entire spectrum. There are a predominantly a few popular libraries
for handling SSL, XML or SOAP communication, and most of them are integrated
in a certain way in an application framework. However, that makes a consistent
implementation of such a framework harder. Of course, one would gain an awful
lot from the experience of such mature libraries, but they may not be a
prefect fit for the concept that the application framework would like
to convey. Such libraries are often integrated with an adapter pattern to make
them fit within the confinements of the application framework, even though some
concessions may need to be made in the process.

In this aspect, \MUDLib serves the purpose foremost, to present yet another
view of the current technologies and how they can be applied in application
development. It is advocating a certain viewpoint on how different
technologies can operate together, with the same underlying concepts and
methodologies, while providing a consistent application framework on all
levels.  Therefore, it is aiming to implement many of the technologies on a
\textit{per need} basis. It is not trying to be complete in every aspect it
supports, but only implements those aspects that are vital for a certain
application. This library is therefore ever evolving!

\chapter{Introduction}
\label{chp:core-introduction}

Some concepts and types are core to a number of technologies and are captured
in the core part of \MUDLib. The core library is not depending on any other
library, other than the standard \Cpp and the Standard Template Library.

The types defined in this library are often used as foundation blocks for
other parts of the \MUDLib library. For example, the concept of a reference
to a computer resource (known as a \textit{handle}) is defined in an abstract
way in the core library, but defined more explicitely in other parts of the
library. The I/O library defines handles to pipes and sockets, while the
event library uses the abstract notion of handles and their concrete
implementations to wait for any specific activity happening on them.

Other parts of the core library are not necessarily used as foundation blocks
for other parts, but contain basic functionality that is not necessarily
bound to a specific domain. For example, the concept of \textit{endianness}
is tied to the underlying computer architecture and is predominantly used in
networking and messaging protocols, but not limited to those.

\chapter{Exceptions}
\label{chp:core-exceptions}

Exceptions are used in the \MUDLib library for signalling an erroneous
condition. Although exceptional behaviour does not necessarily mean conditions
that rarely occur, they can be a useful tool in situations where certain
error conditions are excepted. For example, an application may choose to raise
an exception when a connection cannot be established to a remote machine.
This in itself is not a rare occasion, but can actually be expected at times.
It just depends on what the application does when such an exception is caught
and if particular handling of the exception is required without aborting the
application. The most crude way of handling exceptions is to terminate the
application, but that is a very disastrous method of dealing with such a
scenario. Rather, by carefully planning and handling exceptions, it may trigger
an attempt to reconnect instead. 

The \MUDLib library takes the approach that exceptions can be thrown only
for error conditions, whitout imposing any particular handling on the usage
of this library. Note that not only \MUDLib can throw exceptions, but the
underlying \Cpp library that is used by \MUDLib is also able to throw
certain exceptions. These are not always mentioned in the documentation, but
\MUDLib specific exceptions are documented where they can the thrown and
under which condition.

\section{Exception Classes}
\label{sct:core-exception-classes}

All the \MUDLib exceptions are derived from the \Cpp \indextt{std::exception},
but exceptions defined in a particular \MUDLib library are always associated
to the context of that library. The exceptions defined in the core library
are listed in this section.

\subsection{\texttt{not\_owner}}

The \indextt{not\_owner} exception is defined as

\begin{lstlisting}[style=definition]
// namespace mud::core
class not_owner;
\end{lstlisting}

The \indextt{not\_owner} exception is used to indicate that an operation is
attempted on a resource which is currently not the \textit{exclusive} owner
of that resource. Exclusive ownership is often represented through a
\texttt{unique\_ptr}, where a \texttt{nullptr} value of the underlying
pointer indicates that the user is currently not the owner. This exception can
thus to be thrown in situations like in listing~\ref{lst:not_owner}.

\begin{lstlisting}[%
  style=snippet,
  caption={The \texttt{not\_owner} exception.},
  label=lst:not_owner]
if {ptr == nullptr} {
    throw mud::core::not_owner();
}
\end{lstlisting}

\chapter{Factory}
\label{chp:core-factory}

A factory is a common pattern to create instances of a class based on a
specific key. All the concrete classes registered to the factory are derived
from a common abstract class. The definition of the factory is

\begin{lstlisting}[style=definition]
// namespace mud::core
template<typename AbstractKey, class AbstractClass, class... Args>
class factory;
\end{lstlisting}

The \texttt{AbstractKey} is the type of the key that is used to differentiate
the concrete implementation. This can be an enumeration type or a 
 \texttt{std::type\_index} type.a The \texttt{AbstractClass} is the base class
of all concrete implementations. 

The enumeration can create an associated concrete class (i.e. the concrete class
is not known, but the identifying key is). While the \texttt{std::index\_type}
creates the object directly by the type of the class. Any additional argument
types that needs to be passed to the concrete implementation creator functions
are defined by the \texttt{Args} type.

Each concrete implementation that is registered within the factory declaring
an instance of the \indextt{registrar} type:

\begin{lstlisting}[style=definition]
// namespace mud::core 
template<AbstractKey ConcreteKey, class ConcreteClass>
class factory::registrar;
\end{lstlisting}

This declaration ties a particular \texttt{ConcreteKey} to the instantiation of
a \texttt{ConcreteClass}. Whenever a new instance of an object associated to a
\texttt{ConcreteKey} is to be created by the factory, the \texttt{create} method
if used on the factory:

\begin{lstlisting}[style=definition]
// namespace mud::core 
template<typename AbstractKey, class AbstractClass, class... Args>
std;:unique_ptr<AbstractClass> create(AbstractKey key, Args... args);
\end{lstlisting}

If the \texttt{key} in the factory corresponds to an associated registrar, it
will create a new object of the associated \texttt{ConcreteClass} and return it.
Any arguments that are defined by the \texttt{Args...} argument shall be passed
to the constructor of the \texttt{ConcreteClass}. An example of additional
arguments is presented in section~\ref{sct:factory-class}.

\section{Factory by enumeration type}
\label{sct:factory-enumeration}

When a factory is associated to an enumeration of values, this enumeration is
defined as the \texttt{AbstractKey} of the factory. All registered enumerated
values in the factory will need to declare the \texttt{registrar} instance that
ties the enumerated value to a concrete class.

For example, when modeling a basic shape type and three different shapes, an
enumeration is used to define which type we need to create:

\begin{lstlisting}[style=snippet]
// Enumeration of shapes (the abstract key and concrete key values)
enum class shape_t {
  RECTANGLE,
  CIRCLE,
  TRIANGLE
};

// The base class of any shape (the abstract class)
class Shape {};

// Three defined shapes (the concrete class)
class Rectangle: public Shape {};
class Circle:    public Shape {};
class Triangle:  public Shape {};

// The factory to create shapes based on their type
typedef mud::core::factory<shape_t, Shape> ShapeFactory;
\end{lstlisting}

As the factory does not define any additional arguments in the \texttt{Args...}
argument, each instance will be created using the default constructor.
In order for the type to know which shapes are available, each of these types
need to have a \texttt{registrar} declared in a \Cpp source file that will
automatically register the enumerated value to the corresponding shape:

\begin{lstlisting}[style=snippet]
// Registrations into the factory
ShapeFactory<shape_t::RECTANGLE, Rectangle> rectangle_registrar;
ShapeFactory<shape_t::CIRCLE,    Circle>    circle_registrar;
ShapeFactory<shape_t::TRIANGLE,  Triangle>  triangle_registrar;
\end{lstlisting} 

When this has been set up, the factory can be used to create any shape through
its enumerated value. The following example will create a new circle shape:
\begin{lstlisting}[style=snippet]
auto shape = ShapeFactory::create(shape_t::CIRCLE);
\end{lstlisting} 

\section{Factory by class type}
\label{sct:factory-class}

The previous section uses an enumeration to create the mapping to a concrete
class. This can be beneficial in cases where the actual underlying type is
not known and where the enumeration value (or any integer value) would Be
sufficient. It is also possible to use the \indextt{typeid} of the concrete
class as a way to create an instance of such a type.

For example, when modeling a factory of pets, we define the base class of
a pet and three different kind of pets. As is customary, we give each pet
a name when it is being created, which means we need to pass an additional
argument to constructor of each pet. This also needs to be used in the
pet factory through the definition of a \texttt{Args...} argument type:

\begin{lstlisting}[style=snippet]
// The base class of any pet (the abstract class)
class Pet {};
  
// Three defined shapes (the concrete class)
class Cat: public Pet {
public:
    Cat(const std::string& name);
};

class Dog: public Pet {
public:
    Dog(const std::string& name);
};

class Rabbit: public Pet {
public:
    Rabbit(const std::string& name);
};
  
// The factory to create pets
typedef mud::core::factory<std::type_index, Shape, const std::string&> PetFactory;
\end{lstlisting}

Each of these types need to have a \texttt{registrar} declared in a \Cpp
source file that will automatically register the class type to the
corresponding pet:

\begin{lstlisting}[style=snippet]
// Registrations into the factory
PetFactory<typeid<Recatngle>, Rectangle> rectangle_registrar;
PetFactory<typeid<Circle>,    Circle>    circle_registrar;
PetFactory<typeid<Triangle>,  Triangle>  triangle_registrar;
\end{lstlisting}

When this has been set up, the factory can be used to create any pet through
its \texttt{typeid} while passing the pet's name as an argument:
\begin{lstlisting}[style=snippet]
auto pet = PetFactory::create(typeid<Cat>, "Fluffy");
\end{lstlisting}

\chapter{Polymorphic Containers}
\label{chp:polymorphic containers}

The containers provided by the \Cpp Standard Template Library allow entries
that are all of the same type. If a container is required that can hold
objects of polymorphic types, then we need to resort to defining the containers
as pointers to the polymorphic base class. This is often accomplished with the
use of smart-pointers, like \indextt{shared\_ptr}. If we were to try to define
a container without the use of pointers, and use the base polymorphic type
instead, then the container becomes subject to object splicing~\index{object
splicing}.

Uaing smart-pointers with STL containers does require some more syntactical
sugar as we often need an additional level of indirection when accessing its
elements. For example, if we have the following model of a shapes:

\begin{lstlisting}[style=snippet,
  caption={Polymorphic sample structure.},
  label=lst:polymorphic-sample]
// The base class of any shape
class Shape {};

// Three defined shapes types
class Rectangle: public Shape {};
class Circle:    public Shape {};
class Triangle:  public Shape {};
\end{lstlisting}

Then, a vector of any of these shapes can be constructed as follows. Note
the extra complexity when creating these objects by explicitely allocating
these objects through a smart pointer, and the extra indirection required
when dealing with iterators\footnote{When using range-based loops, the
indirection is not as explicit.}.

\begin{lstlisting}[style=snippet]
typedef std::vector<std::shared_ptr<Shape>> Shapes;

Shapes shapes;
shapes.push_back(std::make_shared<Rectangle>());
shapes.push_back(std::make_shared<Circle>());
shapes.push_back(std::make_shared<Triangle>());
for (auto iter = shapes.cbegin(); iter != shapes.cend(); ++iter)
{
    std::cout << (*iter)->type() << std::endl;
}
\end{lstlisting}

But, more importantly, using the concept of smart-pointers for the sole purpose
of holding objects or polymorphic types, it changes the concept of object
ownership and possible undesirable effects. Take forexample the following code
snippet:

\noindent
\begin{minipage}{\linewidth}
\begin{lstlisting}[style=snippet]
auto object = *shapes.begin();
shapes.clear();
\end{lstlisting}
\end{minipage}

After having retrieved the reference of the object at the start of the
container, the container is cleared. The ownership of all the objects is
supposed to be the container itself and therefore, it is expected that all
the objects in the container to be destroyed. However, since the objects
are smart-pointers, we still have the \texttt{object} pointing to a valid
shape.

The polymorphic containers are resolving this issue by still allowing the
objects to be stored in the container in a polymorphic way, but not allowing
access to the smart-pointer of the objects. The objects are only available
as a pure reference type. The implementation is still based on using smart
pointers, but they are not available as such.

\section{Polymorphic Vector}
\label{sct:poly_vector}

The polymorphic vector allows the storage of polymorphic types (class types
that share a common base class) in an unsorted vector-like container. The
definition of the polymorphic vector is

\begin{lstlisting}[style=definition]
// namespace mud::core
template<class Type, class Allocator = std::allocator<std::shared_ptr<Type>>>
class poly_vector;
\end{lstlisting}

where the \texttt{Type} resembles the base-type. All objects that are
maintained in this container need to be of this type, or a derivative thereof.
The objects are stored internally using a \texttt{std::shared\_ptr} but that
is never exposed publicly.

For the remainder of this section, consider the polymorphic sample hierarchy
from listing~\ref{lst:polymorphic-sample}. As the base class is of type
\texttt{Shape}, an empty vector of shapes will be defined as

\begin{lstlisting}[style=snippet]
poly_vector<Shape> shapes;
\end{lstlisting}

\subsection{Construction and Destruction}

A polymorphic vector can be constructed in number of ways. These include the
construction of an empty vector, copy from another vector, or moving the
contents from one vector to another:

\begin{lstlisting}[style=definition,numbers=left]
poly_vector();
poly_vector(const poly_vector& rhs);
poly_vector(poly_vector&& rhs);
poly_vector& operator=(const poly_vector& rhs);
poly_vector& operator=(poly_vector&& rhs);
virtual ~poly_vector();
\end{lstlisting}
\begin{enumeratedef}
  \item The default constructor, creating an empty container.
  \item Constructing a new vector by copying the contents of another vector.
        Note that the copied items are \emph{not} deep-copied, but their
        underlying smart-pointers are copied instead. This in effect shares
        the same object between two vectors.
  \item Constructing a new vector by moving the contents from another vector.
        The vector \texttt{rhs} will be empty.
  \item Copy the contents of another vector to this vector. All existing will
        be removed prior to copying. Note that the copied items are
        \emph{not} deep-copied, but their underlying smart-pointers are copied
        instead. This in effect shares the same object between two vectors.
  \item Move the contents of another vector to this vector. The vector
        \texttt{rhs} will be empty.
  \item Destructor that destroys all containing elements.
\end{enumeratedef}

\subsection{Manipulation and Access}

\subsubsection{Inserting elements}

Modifying the polymorphic vector involves the direct manipulation of the
contents of the vector by inserting new objects or removing objects that are
referenced by an iterator. The inserting semantics of a polymorphic vector
differs from the Standard Template Library vector as the correct polymorphic
type or any of its derivatives need to be defined. However, this is simplified
by defining templated inserting functions. The following set of insertion
operations are available:

\begin{lstlisting}[style=definition,numbers=left]
template<class PolyType> void push_back(const PolyType& value);
template<class PolyType> void push_back(PolyType&& value);
template<class PolyType> void insert(const_itrator pos, const PolyType& value);
template<class PolyType> void insert(const_itrator pos, PolyType&& value);
\end{lstlisting}
\begin{enumeratedef}
  \item Insert a new object of type \texttt{PolyType} at the end of the vector
        by copying the \texttt{value}.
  \item Insert a new object of type \texttt{PolyType} at the end of the vector
        by moving the \texttt{value}.
  \item Insert a new object of type \texttt{PolyType} at the \texttt{pos}
        position by copying the \texttt{value}.
  \item Insert a new object of type \texttt{PolyType} at the \texttt{pos}
        position by moving the \texttt{value}.
\end{enumeratedef}

The moving semantics should be considered if the object being moved into the
container is no longer required. The following code snippet shows the typically
usage of these operations.

\begin{lstlisting}[style=snippet]
// (1) Copy a rectangle at the end of the vector.
Rectangle rect;
shapes.push_back(rect);

// (2) Move a rectangle at the end of the vector.
Rectangle rect;
shapes.push_back(std::move(rect));
shapes.push_back(Rectangle());

// (3) Copy a rectangle at a certain position of the vector.
Rectangle rect;
poly_vector<Shape>::const_iterator pos = shapes.begin();
shapes.insert(pos, rect);

// (4) Move a rectangle at a certain position of the vector.
Rectangle rect;
poly_vector<Shape>::const_iterator pos = shapes.begin();
shapes.insert(pos, std::move(rect));
shapes.insert(pos, Rectangle());
\end{lstlisting}

The \texttt{emplace} and \texttt{emplace\_back} operations used in the
Standard Template Library cannot be defined in a polymorphic manner as the type
of the object to be inserted is required.

\subsubsection{Removing elements}

Removing elements at certain positions, or removing all elements is
accomplished by using the following modification operations:
\begin{lstlisting}[style=definition,numbers=left]
void clear();
iterator erase(iterator pos);
iterator erase(const_iterator pos);
iterator erase(const_iterator first, const_iterator last);
\end{lstlisting}
\begin{enumeratedef}
  \item[1] Clear the contents of the container.
  \item[2-3] Remove the element at the \texttt{pos} position and return an
           iterator to the next element in the vector. 
  \item[4] Remove the elements from \texttt{first} up to \texttt{last}
           (exclusive) and return an iterator to the \texttt{last} position.    
\end{enumeratedef}

The following code snippet shows the typically usage of these operations.

\begin{lstlisting}[style=snippet]
// (1) Clear the entire container.
shapes.clear();

// (2-3) Erase an element at a certain position.
poly_vector<Shape>::const_iterator pos = shapes.begin();
auto new_pos = shapes.erase(pos);

// (4) Erase a range of elements.
poly_vector<Shape>::const_iterator first = shapes.begin();
++first;
poly_vector<Shape>::const_iterator last = shapes.begin();
--last;
auto new_pos = shapes.erase(first, last);
\end{lstlisting}

\subsubsection{Capacity}

There are two methods available to query the number of elements of the
container:
\begin{lstlisting}[style=definition,numbers=left]
size_t size() const;
bool empty() const;
\end{lstlisting}
\begin{enumeratedef}
  \item Return the number of elements in the container.
  \item Return true if there are no elements in the container.
\end{enumeratedef}

The following code snippet shows the typically usage of these operations.
  
\begin{lstlisting}[style=snippet]
if (!shapes.empty()) {
    std::cout << "The container contains " << shapes.size() << " shapes"
              << std::endl;
}
\end{lstlisting}

\subsubsection{Direct access}

The polymorphic vector has direct access to the elements (other than through
iterators) through a few methods. All return a reference to the object, or
an \texttt{std::out\_of\_range} exception.

\begin{lstlisting}[style=definition,numbers=left]
  reference at(size_type pos);
  const_reference at(size_type pos) const;
  reference operator[](size_type pos);
  const_reference operator[](size_type pos) const;
  reference front();
  const_reference front() const;
  reference back();
  const_reference back() const;
\end{lstlisting}
\begin{enumeratedef}
  \item[1-2] Return a reference to the object at the \texttt{pos} position.
  \item[3-4] Return a reference to the object at the \texttt{pos} position.
  \item[5-6] Return a reference to the first object of the container.
  \item[7-8] Return a reference to the last object of the container.
\end{enumeratedef}

As the reference is a reference to the base \texttt{Type}, it requires
explicit casting to access the object as the appropriate derived type.

\begin{lstlisting}[style=snippet]
// The following are all the same
auto rectangle = dynamic_cast<Rectangle&>(shapes.at(0));
auto rectangle = dynamic_cast<Rectangle&>(shapes.[0]);
auto rectangle = dynamic_cast<Rectangle&>(shapes.front());
\end{lstlisting}

\subsection{Iterators}

Iterators are objects that point to elements within a container. The iterators
of a polymorphic vector are bi-directional, meaning that they can use the
postfix and infix operators to move forwards and backwards in the container.
Where the iterator starts and whether or not an iterator is referencing a
constant or non-constant element is determined by the method used.

\begin{lstlisting}[style=definition,numbers=left]
iterator begin()
const_iterator begin() const;
const_iterator cbegin() const;
iterator end();
const_iterator end() const;
const_iterator cend() const;
reverse_iterator begin();
const_reverse_iterator rbegin() const;
const_reverse_iterator crbegin() const;
reverse_iterator rend();
const_reverse_iterator rend() const;
const_reverse_iterator crend() const;
\end{lstlisting}
\begin{enumeratedef}
  \item[1-3] Return an iterator to the beginning of the container.
  \item[4-6] Return an iterator to the end of the container.
  \item[7-9] Return an iterator to the beginning of the container.
  \item[10-12] Return an iterator to the end of the container.
\end{enumeratedef}

\begin{lstlisting}[style=snippet]
// Iterate from begin to the end
for (auto iter = shapes.begin(); iter != shapes.end(); ++iter) {
}

// Reverse iterator from the end to the beginning
for (auto iter = shapes.rbegin(); iter != shapes.rend(); ++iter) {
}

// Range-based loops
for (auto& item: shapes) {
}
\end{lstlisting}

\section{Polymorphic Map}
\label{sct:poly_map}

The polymorphic map allows the storage of polymorphic types (class types
that share a common base class) in a sorted map-like associative container. The
entries are accessible by a key-value pair. The definition of the polymorphic map is

\begin{lstlisting}[style=definition]
// namespace mud::core
template<class Key, class Type, class Compare=std::less<Key>,
         class Allocator = std::allocator<std::pair<const Key, shared_ptr<Type>>>>
class poly_map;
\end{lstlisting}

where the \texttt{Key} is the type of the key to hold an object of type
\texttt{Type}, which resembles the base-type. All objects that are maintained
in this container need to be of this base type, or a derivative thereof.
The objects are stored internally using a \texttt{std::shared\_ptr} but that
is never exposed publicly. The keys are order as specified by the
\texttt{Compare} template argument, which defaults to the standard incremental
ordering of the \texttt{Key} type.

For the remainder of this section, consider the polymorphic sample hierarchy
from listing~\ref{lst:polymorphic-sample}. Assume that the unique key in the
map is an integer value, and the base class is of type \texttt{Shape}, an empty
map of shapes will be defined as

\begin{lstlisting}[style=snippet]
poly_map<int, Shape> shapes;
\end{lstlisting}

\subsection{Construction and Destruction}

A polymorphic map can be constructed in number of ways. These include the
construction of an empty map, copy from another map, or moving the
contents from one map to another:

\begin{lstlisting}[style=definition,numbers=left]
poly_map();
poly_map(const poly_map& rhs);
poly_map(poly_map&& rhs);
poly_map& operator=(const poly_map& rhs);
poly_map& operator=(poly_map&& rhs);
virtual ~poly_map();
\end{lstlisting}
\begin{enumeratedef}
  \item The default constructor, creating an empty container.
  \item Constructing a new map by copying the contents of another map.
        Note that the copied items are \emph{not} deep-copied, but their
        underlying smart-pointers are copied instead. This in effect shares
        the same object between two maps.
  \item Constructing a new map by moving the contents from another map.
        The map \texttt{rhs} will be empty.
  \item Copy the contents of another map to this map. All existing entries will
        be removed prior to copying. Note that the copied items are
        \emph{not} deep-copied, but their underlying smart-pointers are copied
        instead. This in effect shares the same object between two maps.
  \item Move the contents of another map to this map. The map
        \texttt{rhs} will be empty.
  \item Destructor that destroys all containing elements.
\end{enumeratedef}

\subsection{Manipulation and Access}

\subsubsection{Inserting elements}

Modifying the polymorphic map involves the direct manipulation of the
contents of the map by inserting new objects or removing objects that are
referenced by a key. The inserting semantics of a polymorphic map
differs from the Standard Template Library map as the correct polymorphic
type or any of its derivatives need to be defined. However, this is simplified
by defining templated inserting functions. The following set of insertion
operations are available:

\begin{lstlisting}[style=definition,numbers=left]
template<class PolyType> PolyType& insert(const Key& key, const PolyType& value);
\end{lstlisting}
\begin{enumeratedef}
  \item Insert a new object of type \texttt{PolyType} under the \texttt{key}
        by copying the \texttt{value}.
\end{enumeratedef}

\begin{lstlisting}[style=snippet]
// Insert a rectangle with key 100.
Rectangle rect;
shapes.insert(100, rect);
\end{lstlisting}

The \texttt{emplace} operations used in the Standard Template Library cannot
be defined in a polymorphic manner as the type of the object to be inserted is
required.

\subsubsection{Removing elements}

Removing elements at certain positions, or removing all elements is
accomplished by using the following modification operations:
\begin{lstlisting}[style=definition,numbers=left]
void clear();
size_type erase(const Key& key);
iterator erase(iterator pos);
iterator erase(const_iterator pos);
iterator erase(const_iterator first, const_iterator last);
\end{lstlisting}
\begin{enumeratedef}
  \item[1] Clear the contents of the container.
  \item[2] Clear the element with the specified \texttt{key} and return the
           number of elements removed (either 0 or 1).
  \item[3-4] Remove the element at the \texttt{pos} position and return an
           iterator to the next element in the map.
  \item[5] Remove the elements from \texttt{first} up to \texttt{last}
           (exclusive) and return an iterator to the \texttt{last} position.
\end{enumeratedef}

The following code snippet shows the typically usage of these operations.

\begin{lstlisting}[style=snippet]
// (1) Clear the entire container.
shapes.clear();

// (2) Remove the element with matching key.
shapes.erase(100);

// (3-4) Erase an element at a certain position.
poly_map<int, Shape>::const_iterator pos = shapes.begin();
auto new_pos = shapes.erase(pos);

// (4) Erase a range of elements.
poly_map<int, Shape>::const_iterator first = shapes.begin();
++first;
poly_map<int, Shape>::const_iterator last = shapes.begin();
--last;
auto new_pos = shapes.erase(first, last);
\end{lstlisting}

\subsubsection{Capacity}

There are two methods available to query the number of elements of the
container:
\begin{lstlisting}[style=definition,numbers=left]
size_t size() const;
bool empty() const;
\end{lstlisting}
\begin{enumeratedef}
  \item Return the number of elements in the container.
  \item Return true if there are no elements in the container.
\end{enumeratedef}

The following code snippet shows the typically usage of these operations.

\begin{lstlisting}[style=snippet]
if (!shapes.empty()) {
    std::cout << "The container contains " << shapes.size() << " shapes"
              << std::endl;
}
\end{lstlisting}

\subsubsection{Direct access}

The polymorphic map has direct access to the elements (other than through
iterators) through a few methods. All return a reference to the object, or
an \texttt{std::out\_of\_range} exception.

\begin{lstlisting}[style=definition,numbers=left]
  reference at(const Key& key);
  const_reference at(const Key& key) const;
  iterator find(const Key& pos);
  const_reference find(const Key& key) const;
\end{lstlisting}
\begin{enumeratedef}
  \item[1-2] Return a reference to the object matching the \texttt{key}.
  \item[3-4] Return a reference to the object matching the \texttt{key}.
\end{enumeratedef}

As the reference is a reference to the base \texttt{Type}, it requires
explicit casting to access the object as the appropriate derived type.

\begin{lstlisting}[style=snippet]
// The following are all the same
auto rectangle = dynamic_cast<Rectangle&>(shapes.at(0));
\end{lstlisting}

\subsection{Iterators}

Iterators are objects that point to elements within a container. The iterators
of a polymorphic map are bi-directional, meaning that they can use the
postfix and infix operators to move forwards and backwards in the container.
Where the iterator starts and whether or not an iterator is referencing a
constant or non-constant element is determined by the method used.

\begin{lstlisting}[style=definition,numbers=left]
iterator begin()
const_iterator begin() const;
const_iterator cbegin() const;
iterator end();
const_iterator end() const;
const_iterator cend() const;
reverse_iterator begin();
const_reverse_iterator rbegin() const;
const_reverse_iterator crbegin() const;
reverse_iterator rend();
const_reverse_iterator rend() const;
const_reverse_iterator crend() const;
\end{lstlisting}
\begin{enumeratedef}
  \item[1-3] Return an iterator to the beginning of the container.
  \item[4-6] Return an iterator to the end of the container.
  \item[7-9] Return an iterator to the beginning of the container.
  \item[10-12] Return an iterator to the end of the container.
\end{enumeratedef}

\begin{lstlisting}[style=snippet]
// Iterate from begin to the end
for (auto iter = shapes.begin(); iter != shapes.end(); ++iter) {
}

// Reverse iterator from the end to the beginning
for (auto iter = shapes.rbegin(); iter != shapes.rend(); ++iter) {
}

// Range-based loops
for (auto& entry: shapes) {
}
\end{lstlisting}

\chapter{Handles}
\label{chp:handles}

Accessing computer resources from lower-level libraries, often involves
obtaining unique access to that resource and having a certain form of
unique identification. For example, when a TCP socket is being opened, the
process acquiring that resource would have unique access and the resource
is identified within the process by a unique identification, like a socket
file descriptor.

In modern, event-driven programming technologies, such resources are also
used to efficiently wait for any action to occur on the resource. The good
old POSIX \indextt{select} function is able to efficiently wait on a number
of file descriptors at the same time, until one of them signals a certain
action. For example, a socket file descriptors can be registered to wait
until there is something received.

\begin{lstlisting}[style=definition]
// namespace mud::core
class handle;
\end{lstlisting}

The concept of such computer resources is modeled in \MUDLib as a
\indextt{handle}. This base class is common for any resource supported by
the library. As the handle is associated to a single computer resource,
the access to the resource is governed as a \texttt{std::unique\_ptr},
even within the process itself. This allows security about access to the
resource, in particular in a multi-threaded environment. This base class
only defines the basic functions:

\begin{lstlisting}[style=definition]
type_t type();
virtual bool valid() const;
\end{lstlisting}

that return the type of the resource and whether or not the resource is
currently valid. There is only a defined set of types supported and these
include the following:

\begin{table}[h!]
  \begin{center}
    \begin{tabular}{l|p{0.7\textwidth}}
      \textbf{\texttt{handle::type\_t}} & \textbf{Description}  \\
      \hline
      \texttt{NONE} & An unknown type. Do not use.  \\
      \texttt{ATOMIC\_BOOL} & An atomic boolean state.  \\
      \texttt{SELECT} & A handle that can be used in a POSIX \texttt{select}
          statement. These are \texttt{int} values.  \\
    \end{tabular}
    \caption{Supported \texttt{handle} types.}
  \end{center}
\end{table}

The \texttt{handle} class is an abstract class and cannot be constructed
directly. In order to define a concrete representation of a computer resource
handle is defined by instantiating a template of the \indextt{basic\_handle}
template:

\begin{lstlisting}[style=definition]
// namespace mud::core
template<handle::type_t Type, typename Resource, Resource Invalid>
class basic_handle: public handle;
\end{lstlisting}

This template instantiation defines the handle's supported \texttt{Type}, the
type of the underlying \texttt{Resource} (for example \texttt{int} for the
\texttt{SELECT} handle type), and a value of the resource type that is
associated to an \texttt{Invalid} value. Note that the template instantiation
of the supported handle types is already defined in the library and does not
need to be defined anywhere else. The following is therefore only an example
of the pre-defined \texttt{SELECT} type that references POSIX file
descriptors.

\begin{lstlisting}[style=snippet]
typedef basic_handle<handle::type_t::SELECT, int, 0> select_handle;
\end{lstlisting}

When using these handles, it is possible to return the underlying
\texttt{Resource} value of the handle by using the global template function

\begin{lstlisting}[style=definition]
template<typename Resource>
Resource internal_handle(const std::unique_ptr<handle>& handle);
\end{lstlisting}

\section{Self signalling resource}

The handles associated to computer resources are often used in event triggering
mechanisms like POSIX \indextt{select} or Microsoft Windows
\indextt{WaitForMultipleObjects}. Such mechanisms may require a special handle
that can be used to trigger the mechanism in aborting from any waiting condition.
The self-signalling resource handle is derived from the \indextt{signal}
nested class.

\begin{lstlisting}[style=definition]
// namespace mud::core
class handle::signal;
\end{lstlisting}

and support two methods:

\begin{lstlisting}[style=definition,numbers=left]
void trigger();
void capture();
\end{lstlisting}
\begin{enumeratedef}
  \item Trigger the signal such that any event mechanism that has this handle
        registered, would exit any wait condition/
  \item Once triggered, the resource needs to capture the trigger signal.
\end{enumeratedef}

There are two implementations available of the self-signalling trigger and not
every implementation is available with every underlying operating system. The
\indexrm{pipe} implementation is only available with POSIX based operating
systems, like UNIX and MacOS. The UDP \indexrm{socket} based implementation is
available for all operating systems. They are generally be used in waiting
conditions as follows:

\begin{lstlisting}[style=snippet]
// Declaration of a new signalling resource.
select_handle::signal signaller;

// The associated handle can be used in an fd_set used by a 'select'
fd_set readfds;
FD_ZERO(readfds);
FD_SET(internal_handle<int>(signaller.handle()));
...
int result = ::select(max_fd, &readfds, nullptr, mnullptr, nullptr, nullptr);
if (result > 0) {
    if (FD_ISSET(internal_handle<int>(signaller.handle(), &readfds))) {
        signaller.capture();
    }
}

// Any other thread can unblock this 'select' by triggering the handle, this
// will unblock the 'select' and capture the trigger.
signaller.trigger();
\end{lstlisting}

\chapter{Endian}
\label{chp:endian}\index{endian}

A concept that often raises its head when dealing with platform independent
structures and technologies is the notion of endianness. It defines a system's
representation of multi-byte integral values in memory. In \indexit{big-endian}
systems, the most significate byte is stored at the smallest memory address,
while in \indexit{little-endian} systems, the most significant byte is stored
at the largest memory address.

Platform independent networking techologies often opt for a big endian value
representation (also known as \indexit{network order}). In systems whose
architecture is based on little-endian, every representation of a mutli-byte
integral value needs to convert its value when reading from a network protocol
or writing to a network protocol. The \indextt{endian} class can assist in
that conversion.

\begin{lstlisting}[style=definition]
// namespace mud::core
class endian;
\end{lstlisting}

The \texttt{endian} class defines the two types of endian and can return the
endian type of the underlying architecture:

\begin{lstlisting}[style=definition]
enum class endian_t {
  unresolved,
  little,
  big
};
static endian_t native();
\end{lstlisting}
  
The \texttt{endian} class can assess the current architecture to determine
its native endian system. It can also convert any integral type from one
endian syste, to another, including the native one.

\begin{lstlisting}[style=definition]
static uint16_t convert(endian_t from, endian_t to, uint16_t value);
static uint32_t convert(endian_t from, endian_t to, uint32_t value);
static uint64_t convert(endian_t from, endian_t to, uint64_t value);
\end{lstlisting}

As an example, to convert a native integer to a big-endian one:

\begin{lstlisting}[style=snippet]
uint32_t value = 0x12345678;
uint32_t big = endian::convert(endian::native(), endian::endian_t::big, value);
\end{lstlisting}

\chapter{Tasks}
\label{chp:tasks}\index{tasks}

A task is defined as a packaged statement of work that can be executed 
independently and often by another thread. It is often used in combination with
task workers which execute the tasks and task queues which hold tasks that
needs to be executed. With these concepts in place, it can, for example, be
used in event driven applications where the processing of events is captured
in a task and pushed onto a queue. Any worker from a pool of available worker
threads would then be able to pick up any task from the queue and execute it.

\begin{figure}[h]
  \centering
  \begin{tikzpicture}[start chain=1 going right,node distance=-0.15mm]
    \node [name=t,on chain=1] at (-1.5,-.4) {Task $\longrightarrow$}; 
    \foreach \x in {1,2,...,11} {
      \x, \node [draw,on chain=1,scale=1.8] {};
    } 
    \node [name=w,on chain=1] {$\longrightarrow$ Workers};     
    \node at (1.8,-.95) {Queue};
    \draw [->] (w.south) to [out=210,in=-30](t.south);
    \node at (1.8,-2) {Execute};
    \node at (-1.8, -2) {Result};
    \draw [->] (-1.8, -0.8) -- (-1.8,-1.7);
  \end{tikzpicture}
  \caption{Tasks, queues and their workers.}
\end{figure}

The result of the task's execution can be waited on by other threads. That
allows for synchronisation of task exeutions and other interested parties.

\section{Task}

The \indextt{task} template in the \MUDLib library is conceptually identical to
 the standard \texttt{std::packaged\_task}. The template arguments to the task
define the task execution function's arguments and return type:

\begin{lstlisting}[style=definition]
// namespace mud::core
template<class Result, class... Args>
class task<Result(Args...)>;
\end{lstlisting}

When the task is constructed, the definition of the function to be executed
is defined. The type definitions and methods of the \texttt{task} class are
all associated to the operation it should perform:

\begin{lstlisting}[style=definition,numbers=left]
typedef std::function<Result(Args...)> function_type;
void operator()(Args...);
std::future<Result> get_future();
\end{lstlisting}
\begin{enumeratedef}
  \item The type definition of the function that represents the execution of
        the task.
  \item Executing the function that can be run with arguments at the point
        of execution, or (when bound at construction) without arguments.
  \item The future to access the return value once the task has been executed.
\end{enumeratedef}

As the task can be executed at any time, other threads may be able to wait
for its completion through the \texttt{future} object. As an example of
creating a task that multiplies two values as implemented by a task:

\begin{lstlisting}[style=snippet]
// (1) Creating a task and execution with arguments 
task<int(int, int)> multiplier([](int x, int y) {
    return x * y;
});
multiplier(2, 3);
std::cout << multiplier.get_future().get() << std::endl;

// (2) Creating a task and execution with bound arguments 
task<int(int, int)> multiplier(std::bind([](int x, int y) {
    return x * y;
}, 2, 3);
multiplier();
std::cout << multiplier.get_future().get() << std::endl;
\end{lstlisting}

These two examples show the difference of the time that argument values are
being bound to the task. In the first example, the argument values are passed
on at the time of \textit{execution}, while in the second example, they are
passed on at the time of \textit{construction}. The latter approach is in
particular of interest of creating self-contained packaged tasks that can
be executed without requiring any further context.

Note that any exception that is thrown during the execution of the task
will be captured and made available in the result's \indextt{future}
object.

A task simple, self-contained task that requires no arguments and that does
not return a result is also known as a \indextt{simple\_task}:

\begin{lstlisting}[style=definition]
typedef task<void(void)> simple_task;
\end{lstlisting}  

\section{Task Queue}

As the task is a parcel of work that can be executed is wrapped in an object,
they can be passed on and also stored in containers. A specialised container
that can be used with tasks and a pool of task workers that execute them, is
the \indextt{task\_queue}:

\begin{lstlisting}[style=definition]
// namespace mud::core
template<typename Task>
class task_queue;
\end{lstlisting}  

The task queue operates with tasks that share a common task function-type.
Tasks can be pushed and popped as an thread-safe operation and task workers
can wait when a task becomes available:

\begin{lstlisting}[style=definition,numbers=left]
void push(task_type&& tsk);
bool pop(tast_type& tsk);
bool wait_pop(task_type& tsk);
\end{lstlisting}  
\begin{enumeratedef}
  \item Push a taks onto the queue by moving its contents.
  \item Pop a task from the queue and return true if a task had been popped.
  \item Wait until a task becomes available and pop it from the queue. Returns
        true when a task had been popped.
\end{enumeratedef}

A queue that operates on simple tasks is also known as a
\indextt{simple\_task\_queue}:

\begin{lstlisting}[style=definition]
typedef task_queue<void(void)> simple_task_queue;
\end{lstlisting}  

The task queue has a built-in synchronisation that handles the notification to
task workers when a taks becomes available or by requesting all workers to
terminate. The \indextt{task\_queue\_synchronisation} object

\begin{lstlisting}[style=definition,numbers=left]
// namespace mud::core
struct task_queue_synchronisation;
\end{lstlisting}

contains the condition-variable and associated mutex used for thread-safe
synchronisation. If required, this synchronisation object can be returned from
the task queue:

\begin{lstlisting}[style=definition,numbers=left]
const std::shared_ptr<task_queue_synchronisation>& synchronisation();
\end{lstlisting}

\section{Task Workers and their Pool}

A speciailised thread that uses the \indextt{task\_queue} to wait for any
available task and executes them, is a \indextt{task\_worker}:

\begin{lstlisting}[style=definition]
// namespace mud::core
template<typename Task>
class task_worker : public std::thread;
\end{lstlisting}  
  
This task worker operates on a common task function-type and is therefore
also tied to the task queue of a similar type. The task workers thread
function is basically contineously waiting for a task from the task queue
or until it has received a notification from the task synchronisation object
to terminate itself.

A worker that operates on simple tasks is also known as a
\indextt{simple\_task\_worker}:

\begin{lstlisting}[style=definition]
typedef task_worker<void(void)> simple_task_worker;
\end{lstlisting}  

A \indextt{task\_worker\_pool} is a container of task workers that all
operate on a common shared function-type;

\begin{lstlisting}[style=definition]
// namespace mud::core
template<typename Task> 
class task_worker_pool;
\end{lstlisting} 

At construction, the pool can be created by specifiying the associated
task queue it should consume and the number of workers to create. The
default number of workers, if none is supplied by the standard thread
harware concurrency number:

\begin{lstlisting}[style=definition]
task_worker_pool(const std::shared_ptr<task_queue<Task>>&,
                 size_t nr_workers = std::thread::hardware_concurrency());
\end{lstlisting}
 
When the pool is constructed, it does not yet start the worker threads. The
lifecycle of the pool is determined by

\begin{lstlisting}[style=definition,numbers=left]
void initiate();
void terminate();
void wait();
\end{lstlisting}
\begin{enumeratedef}
  \item Start all the task workers in the pool.
  \item Request all task workers to terminate.
  \item Wait for all task workers to have terminated.
\end{enumeratedef}

The \indextt{simple\_task\_worker\_pool} for simple tasks is specially
defined as

\begin{lstlisting}[style=definition]
typedef task_worker_pool<void(void)> simple_task_worker_pool;
\end{lstlisting}

\section{Example of simple task worker pool}

As an example of a simple task worker pool that operates on an associated task
queue, see the listing~\ref{lst:task-pool}.

\begin{lstlisting}[style=snippet,
  caption={Task pool executing simple tasks.}
  label=lst:task-pool]
// Create a queue and the pool with a number of workers.
simple_task_queue queue = std::make_shared_ptr<simple_task_queue>();
simple_task_worker_pool pool(queue);

// Any thread can push tasks, for example
simple_task tsk([](){
   // Execute something on a separate simple_task_worker thread
});
std::future<void> result_future = tsk.get_future();
queue.push(std::move(tsk));

// Threads can wait for the task to be finished through the future
if (result_future.wait_for(std::chrono::seconds(10)) == std::future_status::ready) {
    result_future.get();
}

// Stop the workers in the pool
pool.stop();
\end{lstlisting}

\part{Events}

Building interactive applications, or applications that react on various
input stimuli, are often designed around an event framework. This allows the
application to define event types and what to do when a certain event
eventuates. This can include events like pressing a button on a keyboard,
dragging the mouse cursor, adding and entry into a database or receiving an
HTTP request. Many of these event frameworks are pre-existing and already
provided by the operating system or some other software development kit.

These pre-existing frameworks, however, cater for only a very specific subset
of events, and are often structured differently from each other. An event
framework that listens to UI input uses a different kind of event technology
than listing for an incoming TCP packet. An extra layer of abtraction is
required to shield the various underlying mechanisms and thus presenting a
uniform event mechanism.

The event framework provided by the \MUDLib library aims to provide such
an event mechanism by creating a facade pattern to the various underlying
pre-existing technologies.

% ============================================================================
% CHAPTER
% ============================================================================

\chapter{Event}

At the heart of the event framework is the concept of an abstraction of an
\indextt{event} as an entity that signals a certain condition. This can
be, for example, a kerybaord press, a message being received on a network
socket, a timer being signalled or a file being updated. These events can
potentially be modeled as an event.

\begin{lstlisting}[style=definition]
// namespace mud::event
class event 
{
    enum class signal_type: uint32_t;
    enum class return_type;
    typedef std::function<return_type(void)> function_type;
};
\end{lstlisting}

An event has three key properties that is maintained, the resource associated
to the signalled condition, the action to wait for and a handler to invoke
when the action on the resource has occurred:

\begin{lstlisting}[style=definition,numbers=left]
event(const std::unique_ptr<handle>& handle, signal_type mask, function_type&& handler);
const std::unique_ptr<handle> handle() const;
signal_type mask() const;
function_type handler() const;
\end{lstlisting}
\begin{enumeratedef}
  \item Construction of an event, specifying the resource handle, the signal
        mask to wait for and the handler to invoke.
  \item The computer resource handle associated to this event.
  \item The signal mask that defines the actions to wait for. This is a
        bitwise-or of signal type values.
  \item The handler to invoke when the signalled action occurred.
\end{enumeratedef}

The \indextt{signal\_type} is an enumeration of 32-bit bit-wise values that
can be added together to create a mask. The current values that are supported
are listed in table~\ref{tbl:signal_type}. How these are interpretted exactly
depends on the context of the type of the handle. Not all handles support all
types.

\begin{table}[h!]
  \begin{center}
    \begin{tabular}{l|p{0.65\textwidth}}
      \textbf{\texttt{event::signal\_type}} & \textbf{Description}  \\
      \hline
      \texttt{NONE} & No signal, ie. no bit in the mask is set. \\
      \texttt{READY} & Any signal, ie. all bits in the mask are set.  \\
      \texttt{READING} & The handle is ready for reading.  \\
      \texttt{WRITING} & The handle is ready for writing.  \\
    \end{tabular}
    \caption{The \texttt{signal\_type} options.}
    \label{tbl:signal_type}
  \end{center}
\end{table}

The handler function is a handler function that does not take any input and
returns a \texttt{return\_type}. This returns is an enumerated value and
instructs the event-loop what to do with this event after it has been signalled
and after the handler has been invoked. The values are listed in
table~\ref{tbl:return_type}.

\begin{table}[h!]
  \begin{center}
    \begin{tabular}{l|p{0.65\textwidth}}
      \textbf{\texttt{event::return\_type}} & \textbf{Description}  \\
      \hline
      \texttt{REMOVE} & Remove the event from the event loop.  \\
      \texttt{CONTINUE} & Continue the event in the event loop.  \\
    \end{tabular}
    \caption{The \texttt{return\_type} options.}
    \label{tbl:return_type}
  \end{center}
\end{table}

This allows events that are no longer needed in the event loop to be removed
(for example, when a network connection has been closed) or to be continued
with the same registration in the \texttt{event\_loop}.

The following example presents the construction of an \texttt{event} that
handles signals when there is something to read on the file-descriptor, and the
registration into an event loop. If it read a 'q' character, it would stop.

\begin{lstlisting}[style=snippet]
// Creation of a read-event for a file-descriptor handle.
event ev(fd_handle, event::signal_type::READING, [fd_handle](){
    char ch;
    read(internal_handle<int>(fd_handle), &ch, 1);
    return (ch == 'q' ? event::retyrn_type::REMOVE
                      : event::return_type::CONTINUE);
})

// Register it to the standard event-loop which should be running on a different thread.
event_loop::global().register_handler(std::move(ev));
\end{lstlisting}

When the event-loop detects that there is something ready to be read, the
handler will be invoked. Note that the loop is typically run on a separate
thread, and therefore the event handler is invoked on that same thread.

Any additional object that is required by the handler, should be made available
using techniques, for example by using
\begin{itemize}
  \item Lambda captures.
  \item Argument binding using \texttt{std::bind} at time of construction.
  \item Defining the handler as a class object and pass the \texttt{this} pointer
        as a \texttt{std::bind}.
\end{itemize}

\chapter{Event Loops}
\label{cpt:event-loops}

In an event-driven architecture, the event-loop play a crucial role. Such an
architecture would be able to listen to a number of events, and wait until
some events happen and react on that. The previous chapter introduced the
concept of an event, which defines the set-up for a single event and the
handler to invoke when the event occurs. The event-loop will tie that all
together and efficiently wait for one or more events contineously and invoke
their handlers when required.

\section{Event Mechanisms}

Over time, advances in computer architecture and operating system design have
seen a number of different ways that computer events can be managed and
waited on in an efficient way. These types of mechanisms are tied to the type
of resource and the method of interaction with the mechanism. One of the
earliest mechanisms involves the \indextt{select} operating system call that
enables a number of file descriptors to be examined at the same time. These
file descriptors are integer based handles and can be expressed in the
\MUDLib as a \indextt{select\_handle} (see chapter~\ref{chp:handles}).
Many operating systems, including the most modern ones, support this still.

Other operating systems have implemented their own style, like the Microsoft
Windows \texttt{WaitForMultipleObjects} type of mechanism, which operate
on Windows specific \texttt{HANDLE}'s. Most operating systems would actually
support more than one mechanism, and a particular type of resource may only
be available with a particular type of mechanism.

When we start talking about user-interface mechanisms that deal with user
interface events (button presses, mouse movements etc), then yet another set of
mechanisms can be found, like X11, Cocoa and Win32 just to name a few. It
should be clear that having a \textit{single} event-loop that can deal with
multiple type of handles and their associated but different event mechanisms,
will still need to use these mechanisms and have a way of combining them
into a single overarching wait mechanism.

\section{Event loop}

In modern software architecture, event-loops play a significant role. It allows
the application to be reactive on any event while in the mean-time continue
with other tasks. It also brings a level of abstraction to the application
as a single event-loop can be used for a variety of events. For example, the
POSIX \indextt{select} mechanism allows an application to wait for multiple
file-descriptors with one call. When any of these file-descriptors
detect an event, the application can examine the \texttt{select} structure to
determine which file descriptor has been triggered.

Over time, a number of event-loops have been developed and are widely in use,
each to serve their own purpose or software architecture, and to extend the
same principles in other realms. As \texttt{select} has been widely used to
handle file-descriptors, it is not designed to be used to handle graphical
user interface events (like moving a window or selecting a check-box).

This leads to applications having to implement multiple event-loops, each
with their own architecture and design criteria, at the same time where each
event-loop handles their own specific event.

Taking a holistic approach, an event-loop is nothing more than a mechanism to
register the interest of an event and be signalled when such an event occurs.
It does not matter if that event is a network socket that receives a new
message, or a check-box that has been selected. By hiding the implementation
of these architecture dependent event-loops, it is possible to create a
platform independent event-loop. It also allows abstraction of different
implementation of common technologies. For example, a \textit{pipe}\index{pipe}
can be used for inter- and intra-process communication, but they are defined
differently on UNIX based systems where the pipe is a file-descriptor that uses
the \texttt{select} mechanism, and on Microsoft Windows system that uses a
\indextt{HANDLE} and a \indextt{WaitForMultipleObjects} mechanism to wait for
any data arriving on the pipe. Since the concept of a pipe is already abstracted
behind a \indextt{handle} abstraction, the event-loop can register that
handle using the appropriate mechanism as well. On UNIX system, the pipe handle
is an integer file-descriptor that registers the pipe with  the
\texttt{select} mechanism, while on Microsoft Windows systems, the pipe handle
is a Windows \texttt{HANDLE} and regsiters the pipe with the
\texttt{WaitForMultipleObjects} mechanism.

\section{\texttt{event\_loop}}
\label{sct:event_loop}

The \MUDLib \indextt{event\_loop} is able to use multiple event mechanisms
concurrently and efficiently. Which underlying mechanism is used is determined
by the type of handle, and is not required to be specified when the handler is
registered. For example, a socket handle will be registered with a POSIX
\texttt{select} based mechanism, an X11 mouse handle will be registered with
the X11 mechanism and a pipe handle will be registered by the \texttt{select}
mechanism in a POSIX based environment or a \texttt{WaitForMultipleObjects}
mechanism in a Microsoft Windows based environment.

Once the handle and its handler function has been registered, the event loop
will wait for any action on that handle and invoke the handler function when
the registered action has occurred.

The \indextt{event\_loop} is usually run on its own thread as it executes the
\indextt{run} method, which will only return once the event loop has been
instructed to terminate.

\begin{lstlisting}[style=definition]
// namespace mud::event
class event_loop;
\end{lstlisting}

When an event-loop is created, the POSIX \texttt{select} mechanism is always
loaded. Any additional mechanisms that are required, can be added explicitely
\textit{before} the event-loop is run (see section~\ref{chp:handles} for a
list of possible handle-types. Each handle-type would typically have its own
mechanism.

\begin{lstlisting}[style=definition]
void add_mechanism(handle::type_t);
\end{lstlisting}

Once the event-loop has been configured with the appropriate mechanisms, it
can enter the execution loop.

\begin{lstlisting}[style=definition,numbers=left]
void run();
void register_handler(event&&);
void deregister_handler(event&&);
void terminate();
\end{lstlisting}

The following example shows the set-up of an event-loop that uses the standard
POSIX \texttt{select} and an additional X11 mechanism. As the loop's execution
does not return, it is spawned in its own thread.

\begin{lstlisting}[style=snippet]
event_loop loop;
loop.add_mechanism(handle::type_t::X11);
std::future<void> future = std::async(
  std::launc::async, [&loop]() {
    loop.run();
  }
);
\end{lstlisting}

Now that the event loop is run, any thread can register an event handler for a
certain handle within the loop. The handler instructs the event-loop what it
is expected to do with the event after the handler is returned. It can either
continue with the current event, or it can de-register it.

\begin{lstlisting}[style=snippet]
// Register a handler and a handler function for a 'read' action
loop.register_handler(event(the_handle, []() {
        // Handle the event, e.g. read from a socket.
        return event::return_type::CONTINUE;
    }), event::signal_t::READING);

// Dergister the handler.
loop.deregister_handler(event(the_handle));
\end{lstlisting}

When the event-loop is no longer required it can be terminated in a thread
safe way. This will deregister all handlers and stop the \texttt{event\_loop}'s
execution run loop.

\section{Global event loop}

A single, global event-loop is available as well, which can be accessed through
the \indextt{global} method. Only when this function is called, will the event
loop be created.

\begin{lstlisting}[style=definition,numbers=left]
static event_loop& global();
\end{lstlisting}

This event loop can easily be used as an application's main loop. Since it is
accessible through a static function, any code can have direct access. The
following example is a simplied extract from the TCP server sample of the
\MUDLib library. A TCP listening socket is opened that accepts any incoming
connection on port 80, using the TCP \indextt{acceptor} class (see
section~\ref{chp:network} for more information about networkin I/O
functionality). Once the socket has been registered, the main function enters
the run-loop from which it does not return until the global event loop has
been explicitely terminated. 

\begin{lstlisting}[style=snippet]
int main(int argc, char** argv)
{
    // Set-up a handle and a handler and register it to the global event-loop. For example,
    // a TCP listening socket to start listening for incoming TCP connections.
    tcp::acceptor acceptor(event_loop::global());
    acceptor.open(tcp::endpoint("lcoalhist", 80));
    acceptor.on_accept([]() {
        // Implement what to do when a connection has been accepted.
    });

    // Start the global event loop. This function does not return unless some other thread
    // explicitely temrinates it.
    event_loop::global()::run();
    return 0;
}
\end{lstlisting}

\part{I/O}

The concept of I/O is a fundamental one in application programming. The
majority of applications would have some concept of using I/O for any kind
of communication. This includes communication with hardware, other software
applications or services and interaction with humans. Input can come from
resources like files, keyboard or through an internet socket connection; while
output can be a display, a printer, a file or perhaps a row in a database.

As can be seen, there is a whole variety of I/O and at various levels. The
types of I/O that are dealt with in this layer of the \MUDLib library is the
low-level operating system supported I/O. This is typically I/O that is
supported by, for example, the POSIX standard and would typically require a
method where multiple kernel events can be examined and processed (e.g.
\indextt{select}, \indextt{poll} and \indextt{kevent}).

\chapter{Stream Buffers}
\label{chp:stream-buffers}

Stream buffers are objects that act as a buffer between the application and
the underlying methods of reading and writing to the I/O device. The
functionality provided by the stream buffers is not only buffering, but are
an integral part in the standard stream technologies.

The \MUDLib \indextt{basic\_streambuffer} class is a specialised
\indextt{std::streambuf} that is capable of reading from and writing to a
\indextt{handle}. The underlying implementation that performs the direct access
to the resource is tied to the device I/O operations associated to the type
of the handle. For example, the POSIX handle would usually use the POSIX
\indextt{read} and \indextt{write} routines.

\begin{lstlisting}[style=definition]
// namespace mud::io
class basic_streambuf;
\end{lstlisting}

As the stream buffer is associated to a \texttt{handle}, it is passed as an
argument to the constructor. The constructor also defines the initial size
of the pre-allocated internal buffer and the size of the put-back buffer.

\begin{lstlisting}[style=definition]
basic_streambuf(const std::unique_ptr<handle>& handle, size_f bufsize, size_t putbacksize);
\end{lstlisting}

The \texttt{basic\_streambuf} serves as a base class for concrete
implementations and provides the framework for I/O device mechanisms that
provide methods for reading and writing arbitrary buffers. Any implementation
would require to implement the protected \texttt{read} and \texttt{write}
methods.

\begin{lstlisting}[style=definition,numbers=left]
ssize_t read(void* buffer, size_t count);
ssize_t write(const void* buffer, size_t count);
\end{lstlisting}
\begin{enumeratedef}
  \item Read \texttt{count} bytes into the pre-allocated \texttt{buffer} and
        return the number of bytes read. Return a negative number in case of
        any I/O error and the \texttt{basic\_streambuf} shall raise a
        \texttt{std::system\_error} exception.
  \item Write \texttt{count} bytes from the pre-allocated \texttt{buffer} and
        return the number of bytes written. Return a negative number in case of
        any I/O error and the \texttt{basic\_streambuf} shall raise a
        \texttt{std::system\_error} exception.
\end{enumeratedef}

The data when reading from the device is buffered in an internal buffer. When
the buffer is exhausted and more data is required to be read from the device,
the \texttt{underflow} method is called. If that device is set-up in a
\textit{blocking} mode and there is not enough data, then the call may be
blocked until the data is available, or until the device is no longer accessible.

The same holds for writing to the device. As the data is buffered, the actual
writing to the device may be delayed, for example, when the buffer becomes
full. The writing can be enforced by \textit{flushing} the stream buffer.

The stream buffer is often not used by itself, but through an input and output
stream instead, like \indextt{std::istream} and \indextt{std::ostream}.

class pipe;

\section{Writing to the pipe}

Writing to a pipe can use all the functionality that is provided by the
\indextt{std::ostream} class. This includes formatted stream functionality
and unformatted (binary) output.

\begin{lstlisting}[style=snippet,
  caption={Writing to the pipe},
  label=lst:writing-pipe]
// Create a pipe
mud::io::pipe p;

// Formatted output.
p.ostr() << "Hello" :: std::flush();

// Unformatted output (binary).
uint8_t block[] = { 0x01, 0x00, 0x81, 0xF4 };
p.ostr().write((const char*) block), 4) << std::flush();
\end{lstlisting}

\section{Reading from the pipe}

Reading from a pipe can use all the functionality that is provided by the
\indextt{std::istream}. This includes formatted stream functionality and
unformatted (binary) input.

\begin{lstlisting}[style=snippet,
  caption={Reading from the pipe},
  label=lst:reading-pipe]
// Create a pipe
mud::io::pipe p;

// Formatted input. This may block until data is available.
std::string str;
p.ostr() >> str;

// Unformatted input (binary).
uint8_t block[4];
memset(block, 0, 4);
p.istr().read((char*)block, 4);
\end{lstlisting}

\chapter{Network}
\label{chp:network}

Network programming establishes connection between two network endpoints using
the Open Systems Interconnection (OSI)\index{OSI} model. The OSI model defines
seven layers \cite{ISO7489}, ranging from the physical layer (e.g. ethernet,
wireless, bluetooth, CAN bus) to the application layer (e.g. HTTP, LDAP, SSH).
From a software point of view, up to the transport layer are provided by the
operating system kernel and anything above that needs to be represented by
user-space libraries. Each layer provides the building blocks for the layer
above.

\begin{figure}[hbt]
  \centering

    \bitbox{1}{2: Data Link Layer} \\
    \bitbox{1}{1: Physical Layer}
  \end{bytefield}

  \label{fig:osi}
\end{figure}

\MUDLib provides functionality to support some of the core networking protocols
in these layers and focuses primarily on the protocols that are used in
the Internet Protocol Suite. This includes the widely used \indexrm{Transmission
Control Protocol} (\indexrm{TCP}) and \indexrm{User Datagram Protocol}
(\indexrm{UDP}) protocols defined in the Transport Layer.

\section{Network Layer}

The Network Layer (layer 3 in the OSI model) is responsible for providing
the means to connect transport entities (connection and connection-less) and
to exchange data. The protocol that is widely used in the Internet Protocol
Suite in this layer is the IP protocol, but notable others include ICMP
and IPsec.

\subsection{IP}

The network layer defines (amongst others), the \indexrm{Internet Protocol}
(\indexrm{IP}). This if the principal protocol in the internet protocol suite
and is used to relay data packets across the network boundaries. \MUDLib has
only support for the Internet Protocol version 4 (\indexrm{IPv4}), even-though
version 6 (\indexrm{IPv6}) is becoming more widely spread. Later editions may
include support for version 6.

\subsubsection{Addresses}

In order to transmit packets from one system to another, these systems
needs to be identified such that the packets can be routed successfully.
IPv4 uses a 32-bit address space, conveniently divided into four 8-bit
numbers. This allows for a human representation using
\indexit{dotted-decimal} notation like \texttt{192.168.2.23}, which is
represented by a 32-bit number of \texttt{0xC0A80217}.

The \indextt{ip::address} class is an abstraction of the IPv4 address
representation. Addresses can be specified by using a string representing a
dotted-decimal, or directly as a 32-bit value (in network byte order),
represented by the POSIX \indextt{in\_addr\_t} structure.
Once an address has been created by either methods, the information can
be retrieved in both forms as well.

\begin{lstlisting}[style=definition]
// namespace mud::io::ip
class adress;
\end{lstlisting}

\begin{lstlisting}[style=definition,numbers=left]
address();
address(uint32_t addr);
address(const std::string& addr);
address(const address& rhs);
address& operator=(const address& rhs);
\end{lstlisting}
\begin{enumeratedef}
  \item Specify the 0.0.0.0 IP address.
  \item Specify the IP address as a 32-bit value in network order.
  \item Specify the IP address by dotted-decimal format.
  \item Copy constructor, copying the address from another instance.
  \item Assignment operator, copying the address from another instance.
\end{enumeratedef}

Once the socket has been bound to a local address or when a connection to a
remote peer~\index{peer}\index{socket!peer} has been established, the address
of the local and remote peer is available through the \indextt{source\_address}%
\index{socket!source address} and the \indextt{destination\_address}%
\index{socket!destination address} methods.

\begin{lstlisting}[style=definition,numbers=left]
operator uint32_t() const;
std::string str() const;
\end{lstlisting}
\begin{enumeratedef}
  \item Return the IP address as a 32-bit value in network order.
  \item Return the IP address as dotted-decimal format.
\end{enumeratedef}

\subsubsection{Sockets}

The Network Layer socket consist of the \indextt{basic\_socket} that holds the
reference to the socket resource handle and the \indextt{ip::socket}. These do
not provide the direct functionality of sending and receiving data, but rather
the concepts of representing the Network Layer socket model.

\begin{lstlisting}[style=definition]
// namespace mud::io
class basic_socket;
\end{lstlisting}

The \texttt{basic\_socket} defines the framework for a socket that can be used
on \textit{all} higher layered communication protocols. It does provide the
definition of the socket's communication domain (i.e.  \indexrm{IPv4} and
\indexrm{IPv6}), the type of the socket (i.e. stream, datagram or raw) and
the communication protocol (i.e. \indexrm{TCP}, \indexrm{UDP} and
\indexrm{ICMP}). This combination is often constrained by
the capabilities provided by the underlying platform's operating system.

\begin{lstlisting}[style=definition,numbers=left]
enum class domain_t;
enum class type_t;
enum class protocol_t;
basic_socket(domain_t domain, type_t type, protocol_t protocol);
void close();
\end{lstlisting}
\begin{enumeratedef}
  \item The domain types, see table~\ref{tbl:domain_t}
  \item The socket types, see table~\ref{tbl:type_t}
  \item The protocol types, see table~\ref{tbl:protocol_t}
  \item Open a socket with the defined specifications.
  \item Close the socket.
\end{enumeratedef}

The following communication domain types are supported:
\begin{longtable}{l|p{0.5\textwidth}}
  \textbf{\texttt{basic\_socket::domain\_t}} & \textbf{Description}  \\

\section{Transport Layer} 

The Transport Layer (layer 4 in the OSI model) is responsible for providing
peer-to-peer communications to applications. These communications can be
connection-oriented like \indexrm{Transmission Control Protocol} (\indexrm{TCP})
and connection-less like \indexrm{User Datagram Protocol} (\indexrm{UDP}). These
are the two main protocols at this layer and the only ones that are directly
supported by \MUDLib.

\subsection{TCP}

The TCP\index{TCP} sockets define the connection-oriented TCP protocol, upon
which numerous other protocols are derived from. This level of communication
provides a reliable communication channel between two peers. In order to
establish a connection, one side must be willing to accepting incoming
connection requests (often referred to as the server-side) and this side must
be actively listening for incoming connections before the other side (often
referred to as the client-side) to establish a connection. Once a connection
has been established between two peers, all communication between them is
channeled over a unique connection. The server-side may continue to listen
to additional incoming requests from other clients, and each successful
connection establishment will be given its own unique communication channel.

\subsubsection{Endpoints}

The TCP protocol defines a unique connection by the combination of a host
address and a port number. This combination is represented by the
\indextt{tcp::endpoint} class.

\begin{lstlisting}[style=definition]
// namespace mud::io::tcp
class endpoint;
\end{lstlisting}

The endpoint is used by both the server-side to define the connection paramters
to listen to and for a client-side endpoint to connect to. Once a connection
has been established, a new port is allocated on the server-side to represent
the communication channel that has been established. As a new port is being
used, the original port can still be re-used for accepting additional
client-side connections. 

\begin{lstlisting}[style=definition,numbers=left]
endpont();
endpoint(const ip::address& address, uint16_t port);
endpoint(const endpoint& rhs);
endpoint& operator=(const endpoint& rhs);
\end{lstlisting}
\begin{enumeratedef}
  \item Default constructor, specifying the 0.0.0.0 address and port 0.
  \item Constructor, specifying the IP address and port number.
  \item Copy constructor, copying the address and port from another endpoint
        instance.
  \item Assignment operator, copying the address and port from another endpoint
        instance.
\end{enumeratedef}

The address and port information can be retrieved from the endpoint instance.

\begin{lstlisting}[style=definition,numbers=left]
const ip::address& address() const;
uint16_t port() const;
\end{lstlisting}
\begin{enumeratedef}
  \item Return the IP address.
  \item Return the TCP port number.
\end{enumeratedef}

\subsubsection{Sockets}

The \indextt{tcp::socket} represents each side of this communication channel.
The associated socket resource that uniquely defines the peer's side of the
communication channel is a \indextt{handle} and is owned by the
\indextt{basic\_socket} base class. Only \textit{one} instance of
a socket owns this resource handle. Therefore, it is not possible to copy a
socket to another socket (but they can be referenced or they can be moved to
another socket instance).

\begin{lstlisting}[style=definition]

defined in figure~\ref{fig:mud-io-tcp-class}. Both stream buffers have access

of the data. How the communication protocol is defined, in terms of the
format of the messages, whether or not a peer is expected to send or recieve
data, is out of the scope of this manual and is very much protocol specific.
The classes provided by \MUDLib are just the foundation stones on which the

\end{lstlisting}

  \item Returns the stream object to read from the socket.
  \item Returns the stream object to write to the socket.
\end{enumeratedef}

IP socket options (see section~\ref{tbl:ip-options}).

  \hline
  \texttt{} & \\

As alluded before, the TCP protocol defines one side to be actively listening
while the other side is trying to establish a connection. Both actions are
very specific to either side and are supported by two different helper classes.
The socket classes do not expose the underlying low-level actions that are
commonly seen, like \indextt{bind} and \indextt{listen}. These are abstracted
away behind the server and client-side helper classes.

\subsubsection{Acceptor}

On the server-side, an endpoint is to be allocated to start listening for, and
accepting incoming connection requests. This is handled by the
\indextt{tcp::acceptor} class. This controller class requires a listening
socket resource and an endpoint. Once a connection has been established, a new
socket denoting this new connection is returned from the
\texttt{tcp::acceptor}.

\begin{lstlisting}[style=definition]
// namespace mud::io::tcp
class acceptor;
\end{lstlisting}

The \texttt{acceptor} is used in combination with the event-loop mechanism
provided by the \indextt{event\_loop} (see section~\ref{sct:event_loop}).
When the acceptor has been setup to listen to a certain endpoint, any
incoming connection triggers the registered \indextt{on\_accept} handler
to be called.

\begin{lstlisting}[style=definition,numbers=left]
acceptor(event::even_loop& event_loop = event::event_loop::global());
acceptor(acceptor&& rhs);
acceptor& operator(acceptor&& rhs);
void on_accept(std::function<void(socket&&)> func);
void open(const endpoint& endpoint);
void close();
\end{lstlisting}
\begin{enumeratedef}
  \item Constructor that defines the event loop mechanism to use.
  \item Move constructor that moves the acceptor handling from another
        instance.
  \item Move assignment that moves the acceptor handling from another
        instance.
  \item Register the handler function to invoke when a connection has been
        accepted.
  \item Open the listening socket to start binding and listening on the
        endpoint. Registers the socket with the event-loop.
  \item Remove the underlying listening socket from the event-loop And
        stop accepting any incoming connections.
\end{enumeratedef}

The following example shows the usage of the \texttt{acceptor}. The handler
function is registered before the acceptor is opened.

\begin{lstlisting}[style=snippet]
// Create the acceptor using the global event_loop.
mud::io::tcp::acceptor acceptor();
acceptor.on_accept([](mud::io::tcp::socket&& client) {
    // Connection has been accepted and client socket is available.
}

// Start listening on port 80
acceptor.open(mud::io::tcp::endpoint("127.0.0.1", 80));
\end{lstlisting}

Once the connection has been accepted, the client socket is passed on to the
handler function. This may, for example, pass it on to a \indextt{communicator}
instance (see section~\ref{sct:tcp-communicator}).

\subsubsection{Connector}

Similar to the server-side, the client-side has a controller class that is
responsible for perfoming the connection request. the \indextt{tcp::connector}.
A default socket is used as a placeholder for when the connection has been
established. The \texttt{tcp::connector} will attempt to connect to a specified
\indextt{tcp::endpoint}.

\begin{lstlisting}[style=definition]
// namespace mud::io::tcp
class connector;  
\end{lstlisting}

The \texttt{connector} is used in combination with the event-loop mechanism
provided by the \indextt{event\_loop} (see section~\ref{sct:event_loop}). When
the connector has been set-up to start connecting to a certain peer endpoint,
the established connection triggers the registered \indextt{on\-connect}
handler to be called.

\begin{lstlisting}[style=definition,numbers=left]
connector(event::even_loop& event_loop = event::event_loop::global());
connector(connector&& rhs);
connector& operator(connector&& rhs);
void on_connect(std::function<void(socket&&)> func);
void open(const endpoint& endpoint);
\end{lstlisting}
\begin{enumeratedef}
  \item Constructor that defines the event loop mechanism to use.
  \item Move constructor that moves the connector handling from another
        instance.
  \item Move assignment that moves the connector handling from another
        instance.
  \item Register the handler function to invoke when a connection has been
        accepted.
  \item Open the connecting socket to start binding and connecting to the
        endpoint. Registers the socket with the event-loop.
\end{enumeratedef}

The following example shows the usage of the \texttt{connector}. The handler
function is registered before the connector is opened.
\begin{lstlisting}[style=snippet]
// Create the connector using the global event_loop.
connector connector();
connector.on_connect([](mud::io::tcp::socket&& server) {
    // Connection has been accepted.
}

// Start connecting to port 80
connector.open(mud::io::tcp::endpoint("127.0.0.1", 80));
\end{lstlisting}

Once the connection has been established, the client socket is passed on to the
handler function. This may, for example, pass it on to a \indextt{communicator}
instance (see section~\ref{sct:tcp-communicator}).

\subsubsection{Communicator}
\label{sct:tcp-communicator}

Once a connection has been established, the communication over the socket
connection can be maintained by the \indextt{tcp::communicator} class. It takes
over the ownership of the socket and provides access to the socket's input and
output streams.

\begin{lstlisting}[style=definition]
// namespace mud::io::tcp
class communicator;  
\end{lstlisting}

As is the case with the \texttt{acceptor} and \texttt{connector} classes, the
\texttt{communicator} also uses the event\_loop mechanism. The constructor
itself does not perform any actions yet, as it will open be able to
accomplish that once the communicator has the socket handle assigned to it.

\begin{lstlisting}[style=definition,numbers=left]
communicator(event::even_loop& event_loop = event::event_loop::global());
communicator(communicator&& rhs);
communicator& operator(communicator&& rhs);
void on_receive(std::function<void()> func);
void open(socket&& scoket);
void close();
\end{lstlisting}
\begin{enumeratedef}
  \item Constructor that defines the event loop mechanism to use.
  \item Move constructor that moves the communicator handling from another
        instance.
  \item Move assignment that moves the communicator handling from another
        instance.
  \item Register the handler function to invoke when there is something ready
        on the socket to receive.
  \item Open the communicator to use the supplied socket. Registers the socket
        with the event-loop.
  \item Close the connection.
\end{enumeratedef}

After the \texttt{communicator} has been set-up and registered with the
event-loop, any message that has been received on the socket will trigger
the registered handler.

Any reading and writing action on the socket can be accomplished by using
the two stream methods:

\begin{lstlisting}[style=definition,numbers=left]
std::istream& istr();
std::ostream& ostr();
\end{lstlisting}
\begin{enumeratedef}
  \item Return the stream object to read from the socket.
  \item Return the stream object to write to the socket.
\end{enumeratedef}

As an example, consider a server-side socket that uses a \texttt{communicator}
once a connection has been established. It echoes back any message it receives.

\begin{lstlisting}[style=snippet]
// Class that handles a connected client.
class client
{
public:
    client(tcp::socket&& socket) {
        _communicator.on_receive(std::bind(&client::on_receive, this));
        _communicator.open(std::move(socket));
    }
    ~client() = default;
private:
    void on_receive() {
        std::string msg;
        _communicator.istr() >> msg;
        _communicator.ostr() << msg;
        _communicator.close();
    }
};

// An acceptor handling function that creates a client
std::list<client*> clients;
void on_accept(tcp::socket&& socket)
{
    clients.push_back(new client(std::move(socket)));
}
\end{lstlisting}

\subsection{UDP}

The \indexrm{UDP} socket uses a connection-less protocol to communicate
between two peers. There is no concept of actively accepting and connecting
peers, but messages are just plainly sent and received. As it is a
connection-less protocol, it cannot be directly verified if a message has
actually been received by a peer. Once a message has been received, it is
generally be replied to by using the source address in the UDP packet as
the destination address of the reply.

\subsubsection{Endpoints}

The UDP protocol defines a unique connection by the combination of a host
address and a port number. This combination is represented by the
\indextt{udp::endpoint} class.

\begin{lstlisting}[style=definition]
// namespace mud::io::udp
class endpoint;
\end{lstlisting}

\begin{lstlisting}[style=definition,numbers=left]
endpont();
endpoint(const ip::address& address, uint16_t port);
endpoint(const endpoint& rhs);
endpoint& operator=(const endpoint& rhs);
\end{lstlisting}
\begin{enumeratedef}

\subsubsection{Communicator}
\label{sct:udp-communicator}

Once a socket has been created, the communication over the socket

\end{lstlisting}

The \texttt{communicator} also uses the event\_loop mechanism. The constructor
itself does not perform any actions yet, as it will open be able to
accomplish that once the communicator has the socket handle assigned to it.

\end{lstlisting}

  \item Move constructor that moves the communicator handling from another
        instance.
  \item Move assignment that moves the communicator handling from another
        instance.
  \item Register the handler function to invoke when there is something ready
        on the socket to receive.
  \item Open the communicator to use the supplied socket. Registers the socket
        with the event-loop.
  \item Close the connection.
\end{enumeratedef}

After the \texttt{communicator} has been set-up and registered with the
event-loop, any message that has been received on the socket will trigger
the registered handler.

\begin{lstlisting}[style=definition,numbers=left]

As an example, consider a socket that waits for any messge to be
received on a particular UDP port and uses a \texttt{communicator} for the
communications to echo back the message.

\begin{lstlisting}[style=snippet]

    std::string msg;
    comm.istr() >> msg;
    comm.ostr() << msg;
});

// Create a UDP socket to listen on port 12345

socket.bind(udp::endpoint("127.0.0.1", 12345));

\end{lstlisting}

The event-loop run mechanism is now set-up to wait for any UDP message that
is send on this endpoint.

\part{Protocols}

Protocols are mostly associated to communication between two resources and
define the set of rules and procedures that govern their communication. In
this respect, you would often see protocols being defined in network
communication. For example, the HTTP protocol define the rules and procedures
that are associated between a client device that requests an action to be
performed on a server, and how the server is expected to respond back. If
the client peforms an HTTP GET request (e.g. request the retrieval of a
page in HTML), then the server is expected to respond with the contents or
a defined error code.

Protocols are not limited to networking, but can appear anywhere. Nor are
protocols associated to a particular environment. The same protocol can
be applied in any kind of environment as long as it adheres to the same
rules and procedures, the underlying transport medium may (and thus the
operating environment) may be different.

The protocols defined in \MUDLib are aimed to cerate an abstraction between
the definition of the protocol rules and procedures from the transport
medium. This is often accomplished by implementing the protocols to only
use Standard I/O Stream classes, like \indextt{std::istream} and
\indextt{std::ostream}.

The extensible markup language (\indexrm{XML}) \cite{W3C:XML} is a widely

\chapter{HTML}
\label{chp:html}

The hypertext markup language (\indexrm{HTML}) \cite{HTML} is a widely used
document format and is the World Wide Web's core markup language.
It should be noted that although \indexrm{XML} is similar in its appearance,
it is also stricter in various in its definition. However, from a document
and usage view in \MUDLib, there are many similarities.

\chapter{Gherkin}
\label{chp:test-gherkin}

Behaviour Driven Development~\index{BDD|see {Behaviour Driven Development}}%
\index{Behaviour Driven Development} is a process that encourages software
development to focus on the behaviour of a certain feature. This is
achieved by providing a clear and structured method of describing a feature,
its benefit, the stakeholders, actions and expected outcomes in a
non-technical way:

\begin{itemize}
  \item[] \textbf{Title}: An explicit title
  \item[] \textbf{Narrative}: Description of a feature
  \begin{itemize}
    \item[] \textbf{As a}: Person or role
    \item[] \textbf{I want}: A feature
    \item[] \textbf{So that}: Benefit or value
  \end{itemize}
  \item[] \textbf{Acceptance Criteria}
  \begin{itemize}
    \item[] \textbf{Given}: An initial context
    \item[] \textbf{When}: An action is performed
    \item[] \textbf{Then}: An observable and expected outcome
  \end{itemize}
\end{itemize}

Gherkin~\index{Gherkin} is an adopted language by the wider agile community
which is tightly associated to Behaviour Driven Development. Each feature
would have a set of scenarios, written roughly in the same narrative as
presented above, that define the acceptance criteria of that particular
feature. 

\MUDLib has adopted roughly the same language principles in its the testing
module and has provided the language elements as a set of \Cpp macros. 
These macros are defined. as follows:

\begin{description}
  \item[\indextt{CONTEXT()}] \hfill  \\
        Defines the context that is used by the \texttt{FEATURE}. A constructor
        and destructor can be defined for any additional functionality to
        be executed before and after each \texttt{SCENARIO} test run. The
        name of the context class is \indextt{context}.
  \item[\indextt{END\_CONTEXT()}] \hfill  \\
        Marks the end of a context.
  \item[\indextt{FEATURE(ID)}] \hfill  \\
        Defines the feature by specifying the ID (as a string) and the
        context type that is used by all the scenarios and their steps.
  \item[\indextt{DEFINE\_GIVEN(ID,FUNC)}] \hfill  \\
        Defines a pre-defined \textbf{Given} step with a feature unique ID and
        associated function that sets up the initial context.
  \item[\indextt{DEFINE\_WHEN}] \hfill  \\
        Defines a pre-defined \textbf{When} step with a feature unique ID and
        associated function that performs a particular action.
  \item[\indextt{DEFINE\_THEN}] \hfill  \\
        Defines a pre-defined \textbf{Then} step with a feature unique ID and
        associated function that performs a particular assertion on an
        observable outcome.
  \item[\indextt{END\_DEFINES}] \hfill  \\
        Marks the end of all the predefined steps. This macro should
        \textit{always} be called between the feature and scenario definitions.
  \item[\indextt{SCENARIO(ID)}] \hfill  \\
        Defines a new scenario.
  \item[\indextt{GIVEN(...)}] \hfill  \\
        Defines \textbf{Given} step. If only an ID is specified, then the step
        refers to a pre-defined step. Otherwise it should be provided with a
        function definition. It is used to set-up the initial context of the
        scenario.
  \item[\indextt{WHEN(...)}] \hfill  \\
        Defines \textbf{When} step. If only an ID is specified, then the step
        refers to a pre-defined step. Otherwise it should be provided with a
        function definition. It is used to specify the action to perform.
  \item[\indextt{THEN(...)}] \hfill  \\
        Defines \textbf{Then} step. If only an ID is specified, then the step
        refers to a pre-defined step. Otherwise it should be provided with a
        function definition. It is used to verify the observable outcome by
        using \textbf{ASSERT} macros.
  \item[\indextt{AND(...)}] \hfill  \\
        Defines an additional \textbf{GIVEN}, \textbf{WHEN} or \textbf{THEN}
        specification.
  \item[\indextt{DATA(...)}] \hfill  \\
        Defines a data-table with an (optional) list of column names. If no
        column names are provided, the number of columns are determined by
        the \texttt{ROW} specification. \hfill  \\
        The data table can be provided as part of any \texttt{GIVEN},
        \texttt{WHEN}, \texttt{THEN} and \texttt{AND} and will be only
        acessible within that particular step. The data is available through
        the \indextt{base\_context}.
  \item[\indextt{DATA\_ROW(...)}] \hfill  \\
        Define a row in the data-table.
  \item[\indextt{END\_DATA()}] \hfill  \\
        Marks the end of a data-table.
  \item[\indextt{SAMPLES}] \hfill  \\
        Defines a sample-table with an (optional) list of column names. If no
        column names are provided, the number of columns are determined by
        the \texttt{ROW} specification. \hfill  \\
        The sample-table can be provided as part of a \texttt{SCENARIO}. The
        scenario is run for every sample in the table, and only the contents
        of a particular sample is available though the \indextt{base\_context}.
  \item[\indextt{SAMPLE(...)}] \hfill  \\
        Define a sample in the sample-table.
  \item[\indextt{END\_SAMPLES()}] \hfill  \\
        Marks the end of a sample-table.
  \item[\indextt{END\_FEATURE()}] \hfill  \\
        Marks the end of a feature.
  \item[\indextt{FEATURE\_RUN(...)}] \hfill  \\
        Run all the registered features and their scenarios and return the
        overall result as a pair of \texttt{size\_t} values. The first value
        indicating the total number of scenarios being run, and the second
        value the total number of success. The total number of failed ones can
        be deduced by taking the difference of these two. \hfill  \\
        An optional filter can be supplied to execute only a particular
        feature and/or scenario. This is of the form of
        \begin{description}
          \item \textit{feature} \hfill  \\
                Run all the scenarios of the specified \textit{feature}
          \item \#\textit{scenario} \hfill  \\
                Run all the specified \textit{scenario} in any feature.
          \item \textit{feature}\#\textit{scenario} \hfill  \\
                Run a specific \textit{scenario} in a specific \textit{feature}.
        \end{description}
\end{description}

\noindent
The functions that are provided with the steps is often a lambda function with
the context-type as a reference parameter.

When verifying the expected outcome, a number of assertion macros can be used:

\begin{description}
  \item[\indextt{ASSERT(OUTCOME,EXPECTED)}] \hfill  \\
        Verifies if the outcome is equal to the expected and raises an
        assertion exception if it doesn't.
  \item[\indextt{ASSERT\_THROW(EX,STATEMENTS)}] \hfill  \\
        Verifies if the statements throw an exception of a particular type,
        and raises an assertion exception if it doesn't.
  \item[\indextt{ASSERT\_NO\_THROW(STATEMENTS)}] \hfill  \\
        Verifies if the statements do not throw an exception, and raises an
        assertion exception if it doesn't.
\end{description}

As an example of the usage of the macros, consider listing~\ref{lst:gherkin}.
It defines a simple calculator, that has two variable input field and a
result value. These are defined in the \texttt{context}. This structure is
used as the context-type of the feature and is used with each scenario. Every
invocation of a scenario will use a new copy of the context. This ensures
that there can be no dependency between scenarios, where one scenario is
depending on the result of a previous scenario. Furthermore, the constructor
of the context type can be used to initialise any value prior to the run of
each scenario. And the destructor can be used likewise as the end of each
scenario run to clean-up afterwards.

It is advised that only one feature is defined within one \Cpp source file.
This eliminates compilation and linkage issues when more than one feature
is used. Event though the feature macro will create the same feature class
name, the macro wraps the feature in an unnamed namespace. This ensure that
the visibility is restricted to the internal compilation unit. The same
mechanism can be applied with the context class. If they are wrapped in an
unnamed namespace, other features in different compilation units can still use
the same context class name, but they are treated to be different types.

The calculator has a basic calculation feature that is capable of adding and
multiplying the two input fields. These two actions are separated and
verified by two acceptance criteria, one for addition and one for
multiplication. The example uses predefined \textbf{Given} and \textbf{When}
steps, but they could also be specfied by supplying their details as a
second parameter to the associated steps in the scenario. The verification
of the expected outcome is provided inline with the scenario and asserts the
result field of the context as the observable and testable outcome.

\begin{lstlisting}[style=snippet,
  caption={Gherkin test example},
  label=lst:gherkin]
// The calculator context
CONTEXT()
    int a, b, result;
    std::vector<int> input;
END_CONTEXT()

FEATURE("Basic Calculations")
  // Pre-defined steps
  DEFINE_GIVEN("first value of 2", [](context& ctx) {
        ctx.a = 2; })
  DEFINE_GIVEN("second value of 3", [](context& ctx) {
        ctx.b = 3; })
  DEFINE_WHEN("calculating the sum", [](context& ctx) { 
        ctx.result = ctx.a + ctx.b; })
  DEFINE_WHEN("calculating the product", [](context& ctx) { 
        ctx.result = ctx.a * ctx.b; })
  END_DEFINES()

  SCENARIO("Addition")
    GIVEN ("first value of 2")
      AND ("second value of 3")
    WHEN  ("calculating the sum")
    THEN  ("the sum is 5", [](context& ctx) {
            ASSERT(ctx.result, 5); })

  SCENARIO("Multiplication")
    GIVEN ("first value of 2")
      AND ("second value of 3")
    WHEN  ("calculating the product")
    THEN  ("the product is 6", [](context& ctx) {
            ASSERT(ctx.result, 6); })
\end{lstlisting}

It does become obvious that repeating addition and mulitplication scenarios
with different values can become quite cumbersome. To allow running a
particular multiple times, but with a different set of values every time it
is run, a set of \textit{samples} can be provided. For example, to run
an addition for a number of time, one can write a table as in
listing~\ref{lst:gherkin-samples}. The sample information is accessible through
the \texttt{context::sample} method. This allows the step definitions to access
the actual data of a \textit{(field)} entry by using the templated
\indextt{entry} function:

\begin{lstlisting}[style=definition]
template<typename T> T entry(const std::string& field);
template<typename T> T entry(int field);
\end{lstlisting}

\noindent
There is a version for both named fields and unnamed fields (the latter
is accessible using a zero-based index number). The templated function returns
the field information converted to the desired type \texttt{T}.
 
\begin{lstlisting}[style=snippet,
  caption={Gherkin test example},
  label=lst:gherkin-samples]
SCENARIO("Addition")
  GIVEN ("first value of <first>", [](context& ctx) {
         ctx.a = ctx.sample().entry<int>("first"); })
    AND ("second value of <second>", [](context& ctx) {
         ctx.b = ctx.sample().entry<int>("second"); })
  WHEN  ("calculating the sum")
  THEN  ("the sum is <result>", [](context& ctx) {
          ASSERT(ctx.result, ctx.sample().entry<int>("result")); })
  SAMPLES("first", "second", "result")
      SAMPLE(0, 0, 0)
      SAMPLE(4, 0, 4)
      SAMPLE(0, 4, 4)
      SAMPLE(5, 4, 9)
  END_SAMPLES()
\end{lstlisting}

In a similar fashion, tables of any data can be provided at each \texttt{GIVEN},
\texttt{WHEN}, \texttt{THEN} and \texttt{AND} step. The data is accessible
inside these particular steps only. For example, listing~\ref{lst:gherking-data}
presents an example where a table is used to simulate that a user inputs
a number of values. The data information is accessible through the
\indextt{context::data} method. The data information at a \textit{(row, column)}
can be retrieved through a templated \indextt{entry} function:

\begin{lstlisting}[style=definition]
template<typename T> T entry(int row, const std::string& column);
template<typename T> T entry(int row, int column);
\end{lstlisting}

\noindent
There is a version for both named columns and unnamed columns (the latter
is accessible using a zero-based index number). The templated function returns
the field information converted to the desired type \texttt{T}.

\begin{lstlisting}[%
  style=snippet,
  caption={Gherkin test example},
  label=lst:gherkin-data]
SCENARIO("Average")
  GIVEN ("a list of values", [](context& ctx) {
         for (int i = 0; i < ctx.data().row_count(); ++i) {
             ctx.input.push_back(ctx.data().entry<int>(i, "value"));
         }})
    DATA("value")
        DATA_ROW(5)
        DATA_ROW(6)
        DATA_ROW(7)
    END_DATA()
  WHEN  ("calculating the average", [](context& ctx) {
             int total = 0;
             for(auto value : ctx.input) {
               total += value;
             }
             ctx.result = total / ctx.input.size();
         })
  THEN  ("the average is 6", [](context& ctx) {
          ASSERT(ctx.result, 6); })
\end{lstlisting}

\chapter{Implementation}

The usage of the macros is enough in most cases, and is also encouraged to
do so. However, it may be required to extend the implementation

\section{Exception Classes}
\label{sct:test-exception-classes}

\subsection{\texttt{assertion\_failed}}

The \indextt{assertion\_failed} exception is used to indicate that an test-case
has failed a testable condition. These are captured by the feature test
runner and their details are recorded. These exceptions would therefore not
fail the total execution of the test run, but are used to record that a particular test scenario has failed.

\subsection{\texttt{not\_specified}}

When setting up test scenarios within a feature, it is a common practice to
define elements that are re-used by multiple scenarios only once and let each
scenario reference these generically defined elements. However, when a
reference is not known, it shall raise a \indextt{not\_specified} exception.
These exceptions are captured by the feature test running and their details
are recorded. These exceptions would therefore not fail the total execution of
the test run, but are used to record that a particular test scenario has
failed.

\section{Gherkin classes}

A number of classes are defined, one for each Gherkin item, and include the
\indextt{\_given}, \indextt{\_when} and \indextt{\_then} classes. Each of them
hold a single Gherkin step and have the ability to chain other similar items by
means of the \texttt{And} method (note that \texttt{and} is a keyword in \Cpp,
the method name differs from the coding standard by having a capital first
letter).  The \indextt{\_scenario} and \indextt{\_feature} types refer to the
Gherkin steps and predefined Gherkin steps. 

Note that the naming convention for the Gherkin implementation class names is
to prepend their name with an underscore. This is a signal that the class is
\textit{hidden}, but it does not stop the user from using them.

When creating a new feature (as accomplished through the \Cpp macros), that
feature is defined in a new class that is derived from \texttt{\_feature},
with its constructor populating the predefined Gherkin steps. As the
\texttt{\_feature} is an abstract class (as the \indextt{\_base\_feature} has
an abstract \texttt{run} method), the new feature class will need to implement
the \texttt{run} method. This method should create all the scenarios and run
them, while accumulating their results. 

\section{Feature Factory}

All the defined feature classes are auto-registered to the
\indextt{\_feature\_factory}. A convenience template wrapper is available,
\indextt{\_feature\_registrar}, that can be used as a file-scope global
variable, to facilitate in the registration.

A macro can be used to execute all registered features, or it can be
manually executed through its static \texttt{run} method.
